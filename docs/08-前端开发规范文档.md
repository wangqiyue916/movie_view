# 电影点评系统前端开发规范文档

## 1. 文档目标

本文档用于统一电影点评系统前端开发规范，包括项目结构、命名规则、路由设计、组件拆分、接口请求、权限控制、表单校验、状态管理和页面交互规范。

前端建议技术栈：

- Vue 3
- Vite
- Vue Router
- Pinia
- Element Plus
- Axios

如果实际项目采用 React 或其他框架，可保留规范思想并替换对应技术实现。

## 3. 命名规范

### 3.1 文件命名

| 类型 | 规范 | 示例 |
| --- | --- | --- |
| 页面组件 | PascalCase.vue | MovieDetail.vue |
| 通用组件 | PascalCase.vue | MovieCard.vue |
| API 文件 | camelCase.ts | movieApi.ts |
| Store 文件 | camelCase.ts | userStore.ts |
| 工具文件 | camelCase.ts | dateUtil.ts |
| 常量文件 | camelCase.ts | auditConstants.ts |

### 3.2 变量与方法命名

- 变量、方法使用 camelCase。
- 组件名使用 PascalCase。
- 常量使用大写下划线。
- 布尔变量使用 is、has、can 开头。

示例：

```ts
const movieId = 1
const isLogin = true
const canAudit = false
const AUDIT_STATUS_ONLINE = 'ONLINE'
```

### 3.3 接口字段命名

前端接口字段统一使用 camelCase，与后端 API 文档保持一致。

示例：

```json
{
  "movieId": 1,
  "avgTotalScore": 9.2,
  "createdAt": "2026-07-06 10:00:00"
}
```

## 4. 路由规范

### 4.1 路由结构

```text
/
/login
/register
/movies
/movies/:id
/news
/news/:id
/long-reviews
/long-reviews/:id
/merchandise
/official
/official/certification
/official/submissions
/admin
/admin/audits
/admin/movies
/admin/news
/admin/reports
/admin/users
```

### 4.2 路由命名

路由 name 使用 PascalCase。

示例：

```ts
{
  path: '/movies/:id',
  name: 'MovieDetail',
  component: () => import('@/views/movie/MovieDetail.vue')
}
```

### 4.3 路由权限

路由 meta 中配置权限信息：

```ts
{
  path: '/admin',
  name: 'AdminLayout',
  meta: {
    requiresAuth: true,
    roles: ['ADMIN', 'SUPER_ADMIN']
  }
}
```

前端路由守卫负责体验层拦截，最终权限以后端接口校验为准。

## 5. 页面与组件拆分规范

### 5.1 页面组件

页面组件放在 `views` 目录，负责页面级数据加载、路由参数读取和页面布局。

示例：

- `views/movie/MovieList.vue`
- `views/movie/MovieDetail.vue`
- `views/admin/AuditList.vue`

### 5.2 通用组件

通用组件放在 `components` 目录，负责可复用 UI。

示例：

- `MovieCard.vue`
- `RatingDisplay.vue`
- `AuditStatusTag.vue`
- `PaginationBar.vue`
- `ImageUpload.vue`

### 5.3 组件职责

- 页面组件负责业务编排。
- 通用组件负责展示和交互。
- 不在通用组件里直接写复杂接口请求。
- 表单组件可以通过 props 和 emits 与页面交互。

## 6. API 请求规范

### 6.1 API 文件划分

```text
src/api
  ├── request.ts
  ├── authApi.ts
  ├── movieApi.ts
  ├── ratingApi.ts
  ├── commentApi.ts
  ├── reviewApi.ts
  ├── newsApi.ts
  ├── videoApi.ts
  ├── merchandiseApi.ts
  ├── auditApi.ts
  ├── officialApi.ts
  └── aiApi.ts
```

### 6.2 请求封装

所有接口请求必须通过统一 Axios 实例发送。

统一处理：

- baseURL。
- Token 注入。
- 401 登录失效。
- 403 无权限。
- 后端错误提示。
- loading 状态按页面需要控制。

### 6.3 环境变量

```text
VITE_API_BASE_URL=http://localhost:8080/api
```

禁止在代码中写死后端地址。

## 7. 状态管理规范

建议使用 Pinia。

### 7.1 Store 划分

```text
stores
  ├── userStore.ts
  ├── permissionStore.ts
  ├── appStore.ts
  └── aiStore.ts
```

### 7.2 用户状态

用户 Store 至少保存：

- token
- userId
- username
- nickname
- avatarUrl
- roles
- permissions

退出登录时必须清空用户状态和 Token。

## 8. 权限展示规范

### 8.1 菜单权限

根据角色展示入口：

- 普通用户：前台功能。
- 电影官方用户：电影商后台入口。
- 管理员：管理员后台入口。
- 超级管理员：管理员后台和系统权限入口。

### 8.2 按钮权限

后台按钮根据权限显示：

- 审核通过。
- 审核驳回。
- 上线。
- 下架。
- 删除。
- 封禁用户。
- 分配角色。

按钮隐藏只用于前端体验，后端仍必须校验。

## 9. 表单校验规范

### 9.1 通用校验

所有表单提交前必须进行前端校验：

- 必填项。
- 字符长度。
- 数字范围。
- URL 格式。
- 图片类型和大小。

### 9.2 关键表单规则

| 表单 | 规则 |
| --- | --- |
| 注册 | 用户名、密码、昵称必填 |
| 登录 | 用户名、密码必填 |
| 评分 | totalScore 必填，分值范围合法 |
| 短评 | 内容必填，限制长度 |
| 长评 | 标题、正文、电影 ID 必填 |
| 资讯投稿 | 标题、正文、封面、关联信息按业务要求校验 |
| 周边投稿 | 商品名、图片、电影、类型、外链必填 |
| 驳回审核 | rejectReason 必填 |

## 10. 状态文案统一

### 10.1 审核状态

| 状态 | 前端文案 |
| --- | --- |
| DRAFT | 草稿 |
| PENDING | 待审核 |
| APPROVED | 已通过 |
| ONLINE | 已上线 |
| REJECTED | 已驳回 |
| OFFLINE | 已下架 |
| DELETED | 已删除 |

### 10.2 用户状态

| 状态 | 前端文案 |
| --- | --- |
| NORMAL | 正常 |
| MUTED | 禁言 |
| BANNED | 封禁 |

## 11. 页面交互规范

### 11.1 加载状态

列表、详情、提交按钮应有 loading 状态，避免重复点击。

### 11.2 空状态

列表为空时展示空状态文案，例如：

- 暂无电影数据。
- 暂无相关资讯。
- 暂无待审核内容。

### 11.3 错误提示

接口错误统一使用消息提示。

常见提示：

- 登录已失效，请重新登录。
- 无权限操作。
- 内容不存在或已删除。
- 提交失败，请稍后重试。

## 12. 样式规范

### 12.1 样式组织

全局样式放在 `styles` 目录。

建议拆分：

```text
styles
  ├── reset.scss
  ├── variables.scss
  ├── layout.scss
  └── common.scss
```

### 12.2 页面风格

- 前台页面突出电影内容、海报、评分和资讯。
- 后台页面以表格、筛选、状态标签、操作按钮为主。
- 电影商后台重点展示投稿状态、驳回原因和数据统计。
- 页面按钮、表格、弹窗风格保持统一。

## 13. 前端交付检查清单

- 页面路由是否正确。
- 接口是否统一通过 API 文件调用。
- Token 是否正确携带。
- 401、403 是否统一处理。
- 列表是否分页。
- 表单是否有校验。
- 状态文案是否统一。
- 后台按钮是否按权限显示。
- 空状态、加载状态、错误状态是否完整。
- 是否避免写死后端地址。
