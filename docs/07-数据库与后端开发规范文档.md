# 电影点评系统数据库与后端开发规范文档

## 1. 文档目标

本文档用于统一数据库设计与后端开发之间的落地规范，重点解决命名、实体映射、枚举状态、事务、软删除、分页、权限校验和审核流转等问题。

本规范适用于所有后端模块，包括用户权限、电影、评分短评、长评讨论、首页资讯、视频周边、AI 助手、管理员后台和电影商后台。

## 2. 命名规范

### 2.1 总体原则

- 命名必须清晰表达业务含义，禁止使用无意义缩写。
- 同一概念在数据库、后端代码和接口中必须保持语义一致。
- 数据库字段使用 snake_case。
- 后端类名使用 PascalCase。
- 后端变量、方法、对象字段和 JSON 字段使用 camelCase。
- 枚举值使用大写下划线格式。

### 2.2 数据库命名

| 类型 | 规范 | 示例 |
| --- | --- | --- |
| 表名 | 小写复数名，使用 snake_case | users, movies, long_reviews |
| 字段名 | 小写 snake_case | user_id, movie_id, created_at |
| 主键 | 统一使用 id | id |
| 外键 | 关联表单数名 + _id | user_id, movie_id |
| 时间字段 | 统一使用 _at 结尾 | created_at, updated_at, deleted_at |
| 布尔字段 | 使用 is_ 或 has_ 开头 | is_featured, has_permission |
| 统计字段 | 使用 _count 结尾 | view_count, like_count |
| 状态字段 | 统一使用 status | status |
| 类型字段 | 统一使用 type 或 target_type | product_type, target_type |

数据库命名示例：

```sql
movies
movie_ratings
short_comments
long_reviews
review_replies
news_articles
interpretation_videos
homepage_recommendations
audit_records
```

### 2.3 后端类命名

| 类型 | 规范 | 示例 |
| --- | --- | --- |
| 实体类 | 表业务名 + Entity | MovieEntity, LongReviewEntity |
| DTO | 业务名 + Request 或 DTO | MovieCreateRequest, RatingSubmitRequest |
| VO | 业务名 + VO 或 Response | MovieDetailVO, LoginResponse |
| Controller | 业务名 + Controller | MovieController |
| Service 接口 | 业务名 + Service | MovieService |
| Service 实现 | 业务名 + ServiceImpl | MovieServiceImpl |
| Mapper/Repository | 业务名 + Mapper 或 Repository | MovieMapper |
| 枚举类 | 业务名 + Enum | AuditStatusEnum |
| 异常类 | 业务名 + Exception | BusinessException |

### 2.4 接口字段命名

接口 JSON 字段统一使用 camelCase。

数据库字段到接口字段映射示例：

| 数据库字段 | 后端字段 / JSON 字段 |
| --- | --- |
| movie_id | movieId |
| user_id | userId |
| total_score | totalScore |
| story_score | storyScore |
| visual_score | visualScore |
| acting_score | actingScore |
| created_at | createdAt |
| updated_at | updatedAt |
| reject_reason | rejectReason |
| target_type | targetType |

### 2.5 枚举命名

枚举类使用 PascalCase，枚举值使用大写下划线。

审核状态统一使用：

```text
DRAFT, PENDING, APPROVED, ONLINE, REJECTED, OFFLINE, DELETED
```

用户状态统一使用：

```text
NORMAL, MUTED, BANNED
```

举报状态统一使用：

```text
PENDING, RESOLVED, REJECTED
```

内容类型统一使用：

```text
MOVIE, NEWS, VIDEO, MERCHANDISE, LONG_REVIEW, SHORT_COMMENT, REVIEW_REPLY, OFFICIAL_CERTIFICATION
```

## 3. 数据库与后端实体映射规范

### 3.1 分层对象说明

后端建议区分 Entity、DTO 和 VO。

| 对象 | 用途 | 示例 |
| --- | --- | --- |
| Entity | 与数据库表字段对应 | MovieEntity |
| DTO / Request | 接收前端请求参数 | MovieCreateRequest |
| VO / Response | 返回给前端展示数据 | MovieDetailVO |
| Query | 接收查询条件 | MovieQuery |

要求：

- Entity 不直接作为接口返回对象。
- Controller 不直接操作 Mapper。
- Service 负责业务逻辑、事务、状态流转和权限校验。
- Mapper 或 Repository 只负责数据库读写。

### 3.2 Entity 映射规则

每张业务表对应一个 Entity。

示例：

| 数据库表 | 后端实体 |
| --- | --- |
| users | UserEntity |
| movies | MovieEntity |
| movie_ratings | MovieRatingEntity |
| short_comments | ShortCommentEntity |
| long_reviews | LongReviewEntity |
| news_articles | NewsArticleEntity |
| interpretation_videos | InterpretationVideoEntity |
| merchandise | MerchandiseEntity |
| audit_records | AuditRecordEntity |

### 3.3 DTO 与 VO 使用规则

新增和修改接口必须使用 Request DTO，例如：

```text
MovieCreateRequest
MovieUpdateRequest
RatingSubmitRequest
LongReviewSubmitRequest
AuditRejectRequest
```

列表和详情返回必须使用 VO，例如：

```text
MovieListItemVO
MovieDetailVO
LongReviewDetailVO
NewsArticleVO
MerchandiseVO
```

禁止把数据库密码、删除时间、内部操作字段直接返回给前端。

## 4. 通用字段处理规范

### 4.1 主键规范

- 所有主键统一使用 BIGINT 类型。
- 所有主键字段统一命名为 id。
- 外键字段统一命名为 业务名_id，例如 user_id、movie_id、review_id。

### 4.2 时间字段规范

业务表统一包含：

| 字段 | 说明 |
| --- | --- |
| created_at | 创建时间 |
| updated_at | 更新时间 |
| deleted_at | 删除时间，软删除使用 |

后端新增数据时必须设置 createdAt 和 updatedAt。

后端修改数据时必须更新 updatedAt。

逻辑删除时设置 deletedAt。

### 4.3 创建人与更新人

后台管理、投稿和审核相关表建议记录：

| 字段 | 说明 |
| --- | --- |
| created_by | 创建人 ID |
| updated_by | 更新人 ID |
| submitter_id | 投稿人 ID |
| auditor_id | 审核人 ID |

## 5. 软删除规范

### 5.1 删除策略

业务数据默认使用逻辑删除，不直接物理删除。

逻辑删除规则：

- 删除时设置 deleted_at。
- 查询列表和详情时默认过滤 deleted_at IS NULL。
- 管理后台如需查看删除记录，应单独提供查询条件。

### 5.2 适用表

建议以下表使用软删除：

- users
- movies
- short_comments
- long_reviews
- review_replies
- news_articles
- interpretation_videos
- merchandise

### 5.3 不建议软删除的表

以下表通常作为记录型数据，不建议删除：

- audit_records
- operation_logs
- ai_chat_messages

## 6. 枚举与状态规范

### 6.1 枚举集中管理

审核状态、用户状态、举报状态、内容类型、推荐位类型必须在后端集中定义为枚举类。

禁止在业务代码中散落硬编码字符串，例如：

```text
"PENDING"
"ONLINE"
"REJECTED"
```

应使用枚举值，例如：

```text
AuditStatusEnum.PENDING
AuditStatusEnum.ONLINE
AuditStatusEnum.REJECTED
```

### 6.2 审核状态流转

统一审核状态流转如下：

```text
DRAFT -> PENDING
PENDING -> APPROVED
PENDING -> REJECTED
REJECTED -> PENDING
APPROVED -> ONLINE
ONLINE -> OFFLINE
OFFLINE -> ONLINE
任意后台允许删除状态 -> DELETED
```

### 6.3 状态修改规则

- 投稿人只能执行保存草稿、提交审核、修改被驳回内容。
- 管理员只能通过审核服务执行通过、驳回、上线、下架、删除。
- 前台公开页面只展示 ONLINE 状态内容。
- 管理后台可按权限查看 PENDING、APPROVED、ONLINE、REJECTED、OFFLINE、DELETED 状态内容。

## 7. 通用审核服务规范

### 7.1 审核服务职责

建议后端提供统一审核服务，例如：

```text
AuditService
```

统一处理：

- 查询待审核内容。
- 审核通过。
- 审核驳回。
- 内容上线。
- 内容下架。
- 内容删除。
- 写入审核记录。
- 校验状态是否允许流转。

### 7.2 业务模块对接方式

电影、资讯、视频、周边、长评模块不应各自随意修改审核状态。

推荐对接方式：

1. 业务模块创建内容，状态为 DRAFT 或 PENDING。
2. 管理员调用 AuditService 审核。
3. AuditService 修改目标内容状态。
4. AuditService 写入 audit_records。
5. 前台查询时只读取 ONLINE 状态。

### 7.3 驳回规则

驳回时必须填写 rejectReason。

驳回原因应写入 audit_records，并在投稿人后台或个人中心可查看。

## 8. 事务规范

### 8.1 必须使用事务的场景

以下操作必须使用事务：

- 用户注册并分配默认角色。
- 用户提交或更新电影评分，并更新电影评分统计。
- 点赞或取消点赞，并更新点赞计数。
- 收藏或取消收藏，并更新收藏计数。
- 发布回复，并更新长评回复数。
- 举报内容，并更新举报计数。
- 审核通过、驳回、上线、下架，并写入审核记录。
- 管理员处理举报，并修改目标内容状态。
- 电影商认证审核，并修改用户角色或认证状态。

### 8.2 事务边界

事务应放在 Service 层。

Controller 不处理事务。

Mapper 不处理事务。

### 8.3 事务失败处理

事务内任意一步失败，应整体回滚。

例如评分提交：

1. 新增或更新 movie_ratings。
2. 重新计算电影平均分。
3. 更新 movies 统计字段。

任意一步失败都不能只保存部分数据。

## 9. 分页与查询规范

### 9.1 分页参数

所有列表接口统一使用：

```text
page
pageSize
```

默认值建议：

```text
page = 1
pageSize = 10
```

最大 pageSize 建议不超过 100。

### 9.2 分页返回

统一返回：

```json
{
  "list": [],
  "page": 1,
  "pageSize": 10,
  "total": 0
}
```

### 9.3 查询规则

- 前台公开列表默认只查询 ONLINE 状态。
- 后台列表可以按状态筛选。
- 所有列表必须支持分页。
- 禁止无条件返回全表数据。
- 搜索参数为空时应按默认排序返回分页数据。

### 9.4 排序规则

常见排序字段统一使用：

| 排序字段 | 说明 |
| --- | --- |
| createdAt | 创建时间 |
| publishedAt | 发布时间 |
| viewCount | 浏览量 |
| likeCount | 点赞数 |
| avgTotalScore | 总评分 |
| clickCount | 点击量 |

## 10. 统计字段维护规范

### 10.1 统计字段

常用统计字段包括：

- view_count
- like_count
- favorite_count
- reply_count
- report_count
- click_count
- rating_count
- avg_total_score
- avg_story_score
- avg_visual_score
- avg_acting_score

### 10.2 更新规则

统计字段由后端统一维护，前端不得直接提交统计字段。

新增点赞、取消点赞、发布回复、删除回复、提交评分等操作应同步更新统计字段。

评分均值可以实时计算后写入 movies 表，也可以后续扩展为定时统计。

### 10.3 防重复规则

必须防止重复操作：

| 操作 | 防重复规则 |
| --- | --- |
| 电影评分 | movie_id + user_id 唯一 |
| 点赞 | user_id + target_type + target_id 唯一 |
| 收藏 | user_id + target_type + target_id 唯一 |
| 举报 | 可限制同一用户对同一目标只保留一条待处理举报 |

## 11. 权限与数据归属校验规范

### 11.1 权限校验位置

权限校验必须在后端执行。

前端隐藏按钮只能作为体验优化，不能作为安全依据。

### 11.2 必须校验的内容

后端接口必须校验：

- 用户是否登录。
- Token 是否有效。
- 用户状态是否 NORMAL。
- 用户是否拥有对应角色或权限。
- 用户是否拥有目标数据的操作权限。
- 目标内容状态是否允许当前操作。

### 11.3 数据归属规则

电影官方用户：

- 只能查看和修改自己的投稿。
- 只能重新提交自己的已驳回投稿。
- 不能修改其他电影商内容。
- 不能直接上线内容。

普通用户：

- 只能修改自己的长评草稿或被驳回长评。
- 只能删除或隐藏自己的短评和回复。
- 不能处理他人内容。

管理员：

- 可以处理内容审核和违规内容。
- 普通管理员不能执行超级管理员专属权限。

## 12. 异常与错误码规范

### 12.1 统一异常

建议后端统一封装业务异常，例如：

```text
BusinessException
```

异常返回结构必须与 API 文档一致：

```json
{
  "code": 400,
  "message": "请求参数错误",
  "data": null
}
```

### 12.2 常用错误码

| code | 场景 |
| --- | --- |
| 400 | 请求参数错误 |
| 401 | 未登录或 Token 失效 |
| 403 | 无权限 |
| 404 | 资源不存在 |
| 409 | 状态冲突或重复提交 |
| 500 | 服务器错误 |

### 12.3 常见业务错误

| 场景 | 建议提示 |
| --- | --- |
| 总评分为空 | 总评分不能为空 |
| 重复点赞 | 请勿重复点赞 |
| 内容不存在 | 内容不存在或已删除 |
| 状态不允许审核 | 当前状态不允许执行该操作 |
| 驳回原因为空 | 驳回原因不能为空 |
| 无权修改投稿 | 无权操作该投稿内容 |
| 被封禁用户发布内容 | 当前账号已被封禁 |

## 13. SQL 与索引规范

### 13.1 SQL 编写规则

- 禁止拼接用户输入形成 SQL。
- 模糊搜索参数必须进行安全处理。
- 列表查询必须带分页。
- 查询公开内容必须过滤 status = ONLINE。
- 查询业务表默认过滤 deleted_at IS NULL。

### 13.2 索引使用

高频查询字段应建立索引：

- users.username
- movies.title
- movies.status
- movie_ratings.movie_id
- short_comments.movie_id
- long_reviews.movie_id
- long_reviews.status
- news_articles.status
- interpretation_videos.movie_id
- merchandise.movie_id
- audit_records.target_type + target_id
- homepage_recommendations.section_code

### 13.3 唯一索引

必须设置唯一索引的场景：

- users.username
- roles.code
- permissions.code
- movie_ratings.movie_id + user_id
- likes.user_id + target_type + target_id
- favorites.user_id + target_type + target_id

## 14. 文件与外链校验规范

### 14.1 图片上传

后端必须校验：

- 文件类型。
- 文件大小。
- MIME 类型。
- 文件扩展名。

允许类型建议：

```text
jpg, jpeg, png, webp
```

单文件大小建议不超过 5MB。

### 14.2 外部链接

视频链接和商品购买链接必须校验 URL 格式。

建议只允许 http 和 https 协议。

禁止保存明显危险协议，例如：

```text
javascript:
file:
data:
```

## 16. 模块开发约定

### 16.1 Controller 层

Controller 只负责：

- 接收请求。
- 参数基础校验。
- 调用 Service。
- 返回统一响应。

Controller 不应直接写业务逻辑。

### 16.2 Service 层

Service 负责：

- 核心业务逻辑。
- 权限校验。
- 数据归属校验。
- 状态流转校验。
- 事务控制。
- 调用 Mapper。

### 16.3 Mapper / Repository 层

Mapper 只负责数据库读写。

Mapper 不处理权限、状态流转和业务判断。

## 17. 各模块特别约定

### 17.1 评分模块

- totalScore 必填。
- storyScore、visualScore、actingScore 选填。
- 同一用户对同一电影只能有一条有效评分。
- 重复评分时更新原评分。
- 评分更新后必须同步更新电影平均分。

### 17.2 短评模块

- 短评发布后直接生效。
- 短评不走通用审核流程。
- 短评被举报后进入举报处理流程。
- 管理员可隐藏或删除违规短评。

### 17.3 长评模块

- 长评提交后进入 PENDING。
- 长评审核通过并 ONLINE 后才可公开展示。
- 长评正文使用 Markdown 原文存储。
- Markdown 渲染时前端需要防止 XSS。

### 17.4 资讯、视频、周边模块

- 电影官方用户投稿后进入 PENDING。
- 管理员审核通过并 ONLINE 后才可公开展示。
- 视频和周边点击第三方链接前应记录 clickCount。
- 周边模块不处理订单、支付和库存。

### 17.5 首页模块

- 首页只展示 ONLINE 状态内容。
- 首页推荐位可由管理员手动配置。
- 自动推荐逻辑不得直接修改业务内容本身，只生成展示结果或推荐位配置。

## 18. 开发检查清单

每个后端接口提交前应检查：

- 是否使用统一响应结构。
- 是否使用 camelCase 返回 JSON 字段。
- 是否校验登录状态。
- 是否校验角色权限。
- 是否校验数据归属。
- 是否过滤 deleted_at IS NULL。
- 前台公开接口是否只查询 ONLINE 内容。
- 列表接口是否分页。
- 涉及多表修改是否加事务。
- 状态流转是否通过统一审核服务。
- 是否避免硬编码状态字符串。
- 是否处理重复提交。
- 是否写入必要操作日志或审核记录。
