# 电影点评系统 Git 协作与提交规范文档

## 1. 文档目标

本文档用于统一小组 Git 协作方式，包括仓库结构、分支策略、提交信息、合并流程、冲突处理和代码审核要求，避免多人开发时互相覆盖代码。

## 2. 仓库结构

建议使用一个总仓库管理前后端和文档：

```text
movie-review-system
  ├── movie-web
  ├── movie-server
  ├── docs
  ├── database
  └── README.md
```

如果前端和后端分两个仓库，也应保持相同分支命名和提交规范。

## 3. 分支规范

### 3.1 主分支

| 分支 | 用途 |
| --- | --- |
| main | 稳定版本，用于演示和最终提交 |
| dev | 日常集成分支 |

### 3.2 功能分支

每位成员在自己的功能分支开发：

| 成员 | 分支建议 |
| --- | --- |
| 潘玺名 | feature/user-auth-audit |
| 王琪越 | feature/movie-rating-comment |
| 郭俊岑 | feature/long-review |
| 罗智懿 | feature/home-news |
| 周秋宏 | feature/ai-video-merch |

### 3.3 修复分支

修复问题使用：

```text
fix/问题简述
```

示例：

```text
fix/rating-duplicate-submit
fix/audit-status-error
```

## 4. 开发流程

推荐流程：

1. 从 dev 拉取最新代码。
2. 创建或切换到自己的 feature 分支。
3. 本地开发并自测。
4. 提交代码。
5. 推送到远程 feature 分支。
6. 发起合并到 dev。
7. 负责人检查后合并。
8. dev 稳定后合并到 main。

## 5. 提交信息规范

提交信息格式：

```text
类型: 简短说明
```

常用类型：

| 类型 | 说明 |
| --- | --- |
| feat | 新增功能 |
| fix | 修复问题 |
| docs | 文档修改 |
| style | 样式或格式调整 |
| refactor | 重构 |
| test | 测试相关 |
| chore | 构建、配置或杂项 |

示例：

```text
feat: 添加电影搜索接口
feat: 完成长评详情页
fix: 修复重复评分统计错误
docs: 更新数据库设计文档
style: 调整首页电影卡片样式
refactor: 抽取通用审核服务
```

## 6. 提交粒度

一次提交只做一类事情。

推荐：

- 完成一个接口后提交一次。
- 完成一个页面后提交一次。
- 修复一个明确问题后提交一次。

不推荐：

- 一次提交混合前端、后端、文档、大量无关格式化。
- 提交未说明用途的大量文件。
- 提交本地临时文件。

## 7. 禁止提交内容

禁止提交：

- 数据库真实密码。
- `.env` 本地配置文件。
- IDE 临时文件。
- 编译产物。
- 日志文件。
- 大体积无关文件。

建议 `.gitignore` 包含：

```text
node_modules/
dist/
target/
.idea/
.vscode/
*.log
.env
.env.*
uploads/
```

如需提交环境变量示例，只提交：

```text
.env.example
```

## 8. 合并规范

### 8.1 合并到 dev

合并前必须确认：

- 本地能正常启动。
- 自己负责的接口或页面能正常运行。
- 没有提交真实密码。
- 没有覆盖他人代码。
- 文档或接口变更已经同步说明。

### 8.2 合并到 main

main 分支只保留稳定版本。

合并到 main 前必须经过：

- dev 分支集成测试。
- 核心流程演示通过。
- 数据库脚本可执行。
- 前后端启动说明完整。

## 9. 冲突处理规范

出现冲突时：

1. 先确认冲突文件属于哪个成员负责。
2. 不要随意删除他人代码。
3. 双方一起确认最终内容。
4. 解决冲突后本地运行测试。
5. 再提交冲突解决结果。

高风险冲突文件：

- 路由文件。
- API 请求文件。
- 数据库脚本。
- 权限枚举。
- 审核状态枚举。
- 公共组件。

## 10. 接口变更协作

接口路径、字段、状态枚举变更时，必须同步：

- API 规范文档。
- 前端 API 文件。
- 后端 Controller 和 DTO。
- 联调人员。

禁止只改后端不通知前端，或只改前端不更新文档。

## 11. 数据库脚本协作

数据库变更必须写入 `database` 目录。

建议结构：

```text
database
  ├── schema.sql
  ├── init-data.sql
  └── update
      ├── 001_add_movie_tables.sql
      ├── 002_add_review_tables.sql
      └── 003_add_audit_tables.sql
```

数据库字段变更后必须同步：

- 数据库设计文档。
- Entity。
- Mapper。
- API 返回字段。

## 12. 代码审核清单

合并前检查：

- 是否符合命名规范。
- 是否有无关文件。
- 是否提交真实密码。
- 是否影响其他模块。
- 是否更新必要文档。
- 是否能本地启动。
- 是否有基本自测。
- 是否处理异常情况。

## 13. 小组协作建议

- 每天开发前先拉取 dev 最新代码。
- 每天结束前推送自己的 feature 分支。
- 公共接口、枚举、数据库字段变更要及时同步。
- 先完成基础模块，再做页面美化。
- 答辩前冻结 main 分支，只修关键 bug。
